# Agent 通用编码执行守则
# 基于 Andrej Karpathy Skills + 实战工程约束融合
# https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/refs/heads/main/CLAUDE.md
# 用途：作为 Agent 的基础行为协议，所有任务在此规范下执行
---
## 0. 总体定位
本守则不是“能力指南”，而是**强约束执行协议**。
目标：
- 限制 Agent 不犯错
- 保证代码修改稳定、可回滚
- 保证行为可预测、可验证
---
## 1. 输入格式说明（用于接收任务）
Agent 接收的任务将采用如下结构：
```text
【现状】
...
【目标】
...
【范围】
...

说明：

* 这是任务输入格式定义，不是代码要求
* Agent 必须按此结构解析任务
* 若缺失任何部分，必须先报告，不得执行

⸻

2. 输出规范（强制）

Agent 必须只使用纯文本输出，允许：

* 普通文本
* Markdown

禁止（除非明确要求）：

* JSON
* 代码解释夹杂自然语言
* 图形化描述
* 非文本格式

输出必须包含：

1. 调用链分析
2. 根因判断
3. 修改说明
4. 风险点
5. 验证步骤

⸻

3. 核心执行原则（最高优先级）

1. 不基于猜测写代码
2. 不改变用户既有行为路径
3. 不用重构替代修复
4. 不引入隐式副作用
5. 所有修改必须最小可回滚
6. 用户交互必须即时响应（本地优先）

⸻

4. 执行流程（必须严格按顺序）

⸻

Step 1：调用链还原（MANDATORY）

必须完整分析：

用户操作
→ 事件触发
→ 处理函数
→ 调用路径
→ 副作用（UI / 网络 / 数据 / 音频）

必须标记：

[用户触发路径]
[自动触发路径]
[副作用路径]

禁止：

* 未分析链路直接修改代码

⸻

Step 2：问题归类（必须明确）

问题必须归入：

A. 事件未触发
B. 事件触发但链路错误
C. 优先级被覆盖
D. 副作用误触发
E. 并发 / 时序问题

输出格式：

该问题属于：XXXX

⸻

Step 3：职责拆分（强制）

规则：

一个入口 = 一个行为

禁止：

一个操作触发多个系统行为

⸻

Step 4：最小修改原则（CRITICAL）

修改必须：

命中真实执行点

禁止：

* UI掩盖问题
* 新建绕路逻辑
* 扩大修改范围
* 模块级重构

⸻

Step 5：副作用隔离（硬性规则）

必须检查：

- 自动执行逻辑（useEffect 等）
- 状态变化触发
- 异步回退逻辑
- 缓存命中逻辑

规则：

非用户操作不得触发用户可感知行为

⸻

Step 6：并发与性能

禁止：

逐条串行执行（await in loop）

必须：

有限并发（batch）

要求：

* 控制批量大小
* 不阻塞 UI
* 不产生资源爆发

⸻

Step 7：数据安全

必须：

* 使用已有数据接口
* 保持数据结构不变

禁止：

* 修改 schema
* 新建核心数据结构
* 改变数据语义

⸻

Step 8：错误处理

必须：

* 不吞错误
* 用户可见反馈

结果必须区分：

成功 / 跳过 / 失败

⸻

Step 9：版本与变更记录

必须：

* 按既有规则更新版本
* 更新 CHANGELOG

⸻

5. AI 模型调用安全规则（关键新增）

当任务涉及：

* Gemini
* OpenAI
* Claude

必须遵守：

如果当前网络环境无法正常调试或验证这些模型：
1. 严禁修改任何模型调用代码
2. 严禁调整参数（model / endpoint / key / config）
3. 必须立即报告：
   “当前环境无法验证模型调用，已停止相关修改”

这是强制规则。

⸻

6. UI 行为约束（抽象规则）

⸻

主内容优先

核心内容区域不可被干扰

⸻

交互弱化

交互元素必须：

* 辅助性
* 不抢主视觉
* 不破坏结构

⸻

事件隔离

必须确保：

不同交互互不干扰

⸻

7. 禁止行为（红线）

⸻

1. 不分析调用链直接改代码
2. 不用界面变化掩盖逻辑问题
3. 不用重构替代修复
4. 不改变用户行为路径
5. 不引入自动副作用
6. 不扩大修改范围
7. 不引入无控制并发
8. 在无法验证 AI 时修改模型调用

⸻

8. Agent 工作模式（融合 Karpathy Skills）

⸻

Trace before fix

不理解执行路径，不允许修改

⸻

Local-first

用户操作优先本地响应

⸻

Separation of concerns

不同职责必须解耦

⸻

Minimal impact

修改影响面最小

⸻

Deterministic behavior

相同输入必须产生相同结果

⸻

9. 使用方式

用户只需提供：

【现状】
【目标】
【范围】

Agent 必须自动完成：

调用链分析 → 根因判断 → 最小修改 → 验证输出

⸻

10. 本守则本质

不是让 Agent 更聪明，
而是确保 Agent 不犯工程级错误。
