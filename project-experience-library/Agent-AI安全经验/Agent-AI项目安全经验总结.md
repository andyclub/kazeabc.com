# Agent / AI 项目安全经验总结

> 来源：Andy 经验总结 | 日期：2026-05-06

## 一、核心原则

### 1️⃣ 不要把"隐藏"当安全
**错误认知：**
- IP 没人知道
- 端口没人发现
- 路径比较复杂
- 前端代码压缩了
- 环境变量没公开

**这些都不是安全。**

真正的安全必须建立在：
- 认证（Authentication）
- 授权（Authorization）
- 用户隔离（User Isolation）
- 网络边界（Network Boundary）
- 审计（Audit）

### 2️⃣ 前端永远不是安全边界
前端只能负责：显示、隐藏按钮、页面跳转、体验优化。

真正的权限控制必须在：后端、数据库、网关。

任何浏览器中的逻辑都可以被绕过。

### 3️⃣ 所有公网接口都必须按"敌对环境"设计
公网接口一旦暴露就等于默认会被扫描、重放、伪造请求、批量调用。

因此：
- 所有读写接口默认必须认证
- 所有写接口默认必须限流
- 所有用户数据默认必须隔离

---

## 二、网络安全经验

### 1️⃣ 内部服务默认只监听 localhost
**推荐：** `127.0.0.1` / `localhost`

**避免：** `0.0.0.0`（除非已通过认证网关、TLS、防火墙、限流、日志审计）

### 2️⃣ 不直接暴露内部服务
推荐结构：浏览器 → 认证网关 → 内部 API → 数据库

### 3️⃣ 反向代理不是最终安全
仍然需要：JWT 校验、用户校验、请求大小限制、方法白名单、访问频率限制、日志。

---

## 三、用户数据安全经验

### 1️⃣ user_id 是核心隔离边界
所有用户数据必须围绕 `user_id` 设计。

必须验证：
```
token.user_id == 请求路径 user_id == 写入数据 user_id
```

### 2️⃣ 不相信客户端传来的身份
**禁止：** 前端传什么 user_id 就信什么

**正确做法：** 后端从 JWT/session 中解析真实用户身份

### 3️⃣ 写操作必须可回滚
**危险操作：** 全量覆盖、直接删除、无版本更新

**推荐：** 软删除、版本号、审计日志、历史记录、待确认队列

---

## 四、AI / Agent 安全经验

### 1️⃣ Agent 默认不可信
Agent 本质是高权限、概率型执行体。

Agent 的"执行成功"不等于系统真的安全。必须验证：实际文件、实际日志、实际数据库、实际网络状态、实际测试结果。

### 2️⃣ Agent 不允许直接操作生产环境
**推荐：** 独立 VM、Docker、测试库、测试账号、Mock 数据

**避免：** 生产数据库、真实用户数据、主账号、生产 Token

### 3️⃣ 高自治模式属于高风险模式
自动批准（Always Proceed / Auto Execute / Auto Apply）本质是降低人工审查、提高事故概率。

**只能用于：** 隔离环境
**不能用于：** 核心仓库、生产环境

### 4️⃣ Agent 修改代码前必须理解调用链
**禁止：** 看见报错立即重构、不知道调用路径就改代码

**必须先确认：** 谁触发、谁调用、谁写入、谁依赖、副作用是什么

### 5️⃣ 网络异常时禁止乱改 SDK
**危险行为：** API 不通 → Agent 擅自改 SDK → 改 endpoint → 改认证逻辑 → 引入更大事故

**正确做法：** 先验证网络、DNS、代理、TLS、Key、环境变量，再决定是否修改代码

---

## 五、状态机与任务安全

### 1️⃣ 外部消息不要直接驱动长任务
**危险结构：** Webhook → 直接执行 Agent

**推荐：** Webhook → 任务队列 → Worker → Agent

### 2️⃣ 长任务必须状态化
**推荐状态：** PENDING / RUNNING / SUCCESS / FAILED / RETRY_WAIT / TIMEOUT / CANCELLED

### 3️⃣ 所有自动化任务都必须可观测
必须记录：输入、输出、错误、耗时、重试、最终状态。

否则系统会进入"看起来在运行"但实际上已经死掉。

---

## 六、数据与 AI 成本控制

### 1️⃣ 外部入口不要直接触发 AI
**危险：** 用户请求 → 立即调用大模型 → 不可控成本

**推荐：** 先写入待处理队列，确认后再解析

### 2️⃣ AI 输出必须二次确认
**不要直接：** AI 输出 → 写数据库 → 上线

**推荐：** AI 输出 → 草稿 → 审核 → 确认 → 正式写入

---

## 七、工程安全经验

### 1️⃣ 所有修改必须可回滚
必须保留：git commit、diff、备份、版本号

### 2️⃣ 修复问题优先于重写系统
**危险：** 小问题 → 大重构 → 引入更多问题

**推荐：** 最小修改、最小影响、最小范围

### 3️⃣ 日志比"感觉正常"更重要
不要相信：应该没问题、看起来正常

必须确认：日志、测试、状态、监控

---

## 八、组织级经验

### 1️⃣ 权限最小化
**原则：** 默认不给权限，需要时再授权

**避免：** 万能 Token、共享账号、长期管理员权限

### 2️⃣ 人与 Agent 必须隔离
**不要：** Agent 使用主账号、Agent 持有真实支付权限、Agent 使用生产邮箱

**推荐：** 专用账号、只读权限、子账号、临时 Token

### 3️⃣ 自动化越强，审计越重要
系统自动化提高后，错误传播速度也会提高。

必须强化：日志、审计、回滚、限权、状态机、人工确认点

---

## 九、最终通用原则

1. 默认公网不可信
2. 默认前端不可信
3. 默认 Agent 不可信
4. 默认输入不可信
5. 默认自动化会失败
6. 默认任务会中断
7. 默认日志会缺失
8. 默认用户数据最重要
9. 默认高自治意味着高风险
10. 默认没有审计就等于没有安全
